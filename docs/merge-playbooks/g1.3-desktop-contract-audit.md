# G1.3 Desktop Contract and Renderer Audit

> Date: 2026-05-21
> Branch: integration/platform-convergence-next (58074c5)
> Owner: team-lead (desktop contract lane)

## Findings

### 1. Desktop contract — One shared contract (PASS)

`webui/desktop/shared/desktop-contract.ts` defines:
- `desktopIpcChannels`: IPC channel names (shared)
- `desktopApiPaths`: REST API paths (shared, includes platform-specific paths like `/connect/elevated`)
- `desktopRpcActions`: RPC action names (shared)
- `desktopEventTypes`: SSE event types (shared)
- `desktopServiceCommands`: install/uninstall (shared)
- `desktopDriverInstallTargets`: wintun/tap (Windows-specific, acceptable)
- `desktopRpcErrorCodes`: `helper_unavailable` (shared)

No duplicate `mode`/`session_mode` fields in the contract. The contract uses `mode` consistently.

### 2. VpnStatus interface — `mode` field (PASS)

`webui/src/stores/vpn.ts` defines `VpnStatus.mode` as `'helper' | 'direct' | 'elevated' | 'disconnected'`. No duplicate `session_mode` field exists. The `currentSessionMode` computed property uses `status.value.mode ?? 'helper'` — correct fallback.

### 3. ServiceStatus interface — Matches native DTO (PASS)

`ServiceStatus` in vpn.ts has: `installed`, `running`, `path`, `available`, `mode?`, `endpoint?`, `label?`, `binary_path?`, `service_state?`, `warning?`. This matches the native `ServiceStatusSnapshot` struct in `status_models.hpp` (installed, running, available, mode, path, endpoint, label, binary_path, warning, service_state, has_service_state).

### 4. Structured error handling — Mostly good (PASS with note)

`VpnError` type uses structured fields: `error_type`, `message`, `recoverable`, `recommended_action`. The `normalizeError()` function converts unknown errors to `VpnError` format.

**One concern**: `normalizeError()` line 131 uses `message.includes('elevation_denied')` as a fallback string match to detect elevation_denied errors. This is a brittle pattern — if the native error message changes, this fallback will fail. However, the primary path uses `error_type` from structured responses, so this is only a fallback for unexpected error formats.

### 5. Vue reactive/proxy payloads over IPC — No issue found (PASS)

No evidence of Vue reactive/proxy objects being sent over IPC. The stores use plain data from API responses, not Vue reactive wrappers.

### 6. Platform runners — Correctly separated (PASS)

`webui/desktop/main/platform/` contains:
- `base.ts` — shared base
- `darwin.ts` — macOS-specific (elevated connect/disconnect via AppleScript/osascript)
- `win32.ts` — Windows-specific (elevated connect/disconnect via PowerShell)
- `linux.ts` — Linux stub
- `index.ts` — platform selector
